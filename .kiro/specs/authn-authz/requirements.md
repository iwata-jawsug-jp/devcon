# Requirements Document

## Project Description (Input)

認証・認可(authn/authz)の導入 (Cognito/JWT 等) — GitHub Issue #41 (Epic #46 の子タスク)。

背景: services/backend/python の routers/ は現状無認証。config.py にも認証関連設定が無い。実Webアプリにとって最大の空白で、プロダクト化の前提条件。

目的: APIに認証・認可を導入し、保護されたリソースへのアクセス制御を確立する。

検討・タスク化したい項目:

- 認証方式の選定（例: Cognito + JWT検証、または OIDC / 自前JWT）。CLAUDE.mdでもCognito/JWTを想定として明記済み
- FastAPI側の検証層（Dependsによる現在ユーザ取得、JWKS取得・キャッシュ・署名/aud/iss/exp検証）
- 認可モデル（ロール／スコープ／リソース所有者ベース）の設計
- 既存ルーター（items等）への保護適用と、公開エンドポイント（health）の切り分け
- フロント（services/frontend）のログイン／トークン管理／APIクライアントへの付与
- インフラ: Cognitoユーザープール等のTerraform化（infra/）と、CloudFront/ALB経路での扱い
- シークレット（クライアントシークレット等）はSSM/Secrets Manager管理。VITE_変数には非機密のみ
- 認証/認可のテスト（未認証401・権限不足403・正常系）

完了条件（提案）: 保護対象エンドポイントが未認証で401、権限不足で403を返し、正規トークンで通ること。E2Eでログイン〜保護リソース取得が通る。

注: 影響範囲が広いため、方式選定 → API検証層 → 認可 → フロント → infra の順で個別PRに分割する方針。

リポジトリ構成: モノレポ（services/backend/python: Python/FastAPI, services/frontend: TypeScript/Vue3, infra: Terraform/AWS ap-northeast-1）。ADR-0003によりこの認証機能拡充も既存モノレポ構成内（新規サービス分割なし）で実装する方針が決定済み。

## Introduction

`services/backend/python` の各エンドポイント（`items` の一覧・取得・作成、`health`）は現状すべて無認証で呼び出せる。
`config.py` にも認証関連の設定は無く、`services/frontend` にもログイン機構・認証状態を保持する仕組みは無い。
実プロダクトとして運用するには、保護すべきリソースへのアクセスを認証済みユーザーに限定し、さらに権限に
応じて操作を制限できる必要がある。本機能は、API・Web アプリ双方に認証・認可の仕組みを導入し、未認証
アクセスと権限不足の操作を拒否できるようにする。Epic #46「プロダクト化に向けた機能強化ロードマップ」の
子タスク（Issue #41）であり、モノレポ構成は維持したまま（[ADR-0003](../../../docs/adr/0003-keep-monorepo-through-domain-and-authn-expansion.md)）実装する。

## Boundary Context

- **In scope**: API 保護対象エンドポイントの認証必須化（401判定）、読み取り／書き込みの権限分離による
  403判定、Web アプリのログイン・ログアウト・認証状態表示、未認証状態での保護画面アクセス時の誘導、
  セッション失効時の再認証誘導、認証関連の機密情報の非露出。
- **Out of scope**: 具体的な認証方式・ID プロバイダの選定（設計フェーズで決定する技術選定事項）、
  read/write の二区分を超えるロール・スコープ体系やリソース所有者ベースの認可（例: 自分が作成した
  item のみ操作可能、等）は対象外（Issue #40 のドメインモデリングで扱う）、WAF・レート制限などの
  ネットワーク層の防御（Issue #44 で扱う）、多要素認証・パスワードリセット・セルフサービスアカウント
  管理（本 issue では要望されていない）、独自ドメイン・TLS証明書の整備（別のインフラ課題）。
- **Adjacent expectations**: 既存の `health` エンドポイントは、アプリ外部のインフラ監視（ヘルス
  チェック）から無認証でポーリングされ続けるため、認証必須化の対象から除外され続ける必要がある。

## Requirements

### Requirement 1: 保護対象エンドポイントへの認証の要求

**Objective:** API運用者として、保護対象エンドポイントへのアクセスを認証済みリクエストのみに限定したい。
そうすれば未認証の第三者がリソースを閲覧・操作できない。

#### Acceptance Criteria

1. When 未認証のリクエストが保護対象エンドポイントに送られる、the API shall 401 を返す。
2. When 有効かつ失効していない認証情報を伴うリクエストが保護対象エンドポイントに送られる、the API
   shall そのリクエストを認可判定（Requirement 2）に進める。
3. If リクエストに含まれる認証情報が失効または無効である、then the API shall 401 を返す。
4. The API shall `health` エンドポイントを認証不要のまま維持する。

### Requirement 2: 権限不足時のアクセス拒否

**Objective:** API運用者として、認証済みでも権限が不足するリクエストは拒否したい。そうすれば読み取り
専用の権限で書き込みができてしまう事態を防げる。

#### Acceptance Criteria

1. When 読み取り権限のみを持つ認証済みリクエストが書き込み操作（例: item の作成）を要求する、the API
   shall 403 を返す。
2. When 書き込み権限を持つ認証済みリクエストが書き込み操作を要求する、the API shall 通常の処理を
   継続する。
3. The API shall 認証済みであれば書き込み権限の有無を問わず読み取り操作を許可する。

### Requirement 3: Web アプリでのログイン・ログアウトと認証状態の表示

**Objective:** エンドユーザーとして、Web アプリにログイン・ログアウトし、自分がログイン中かどうかを
把握したい。そうすれば保護された機能に安心してアクセスできる。

#### Acceptance Criteria

1. When 未認証のユーザーが認証を要する画面に遷移しようとする、the Web アプリ shall ログイン手続きへ
   誘導する。
2. When ユーザーがログインに成功する、the Web アプリ shall 元々アクセスしようとしていた画面への
   アクセスを許可する。
3. While ユーザーが認証済みである、the Web アプリ shall 現在ログイン中であることが分かる表示を行う。
4. When ユーザーがログアウトを行う、the Web アプリ shall 認証状態を破棄し、以後のアクセスを未認証
   として扱う。
5. If ログインに失敗する（認証情報が誤っている等）、then the Web アプリ shall エラーを表示し、保護
   された画面へのアクセスを許可しない。

### Requirement 4: セッション失効時の再認証誘導

**Objective:** エンドユーザーとして、操作中にセッションが失効した場合は分かりやすく再ログインを促して
ほしい。そうすれば原因不明のエラーに困惑しない。

#### Acceptance Criteria

1. When 操作中に認証情報の失効により保護対象エンドポイントから401が返る、the Web アプリ shall
   再認証を促す表示を行う。
2. If 再認証が完了していない、then the Web アプリ shall 失効前に取得した保護されたデータを有効な
   ものとして表示し続けない。

### Requirement 5: 認証関連の機密情報の非露出

**Objective:** システム運用者として、認証に関わる機密情報がブラウザやリポジトリに露出しないように
したい。そうすれば認証情報の漏えいを防げる。

#### Acceptance Criteria

1. The システム shall 認証に関わるクライアントシークレットや長期的な資格情報を、ブラウザから参照
   可能な設定やビルド成果物に含めない。
2. The システム shall 認証に関わる機密情報をソースコード管理に含めない（テンプレート／サンプル設定
   のみをコミットする）。
