# ADR-0010: スキャフォールドツールとして copier を採用する

- **Status:** Accepted
- **Date:** 2026-07-14
- **Deciders:** itouhi
- **Related:** #294, [Epic #300](https://github.com/iwata-jawsug-jp/devcon/issues/300)（IDP化ロードマップ
  フェーズ1）、#153（プラットフォームエンジニアリング評価）、#239（共有 Terraform モジュール。
  インフラ層の配布はそちらの担当でありスコープ外）、#298（フェーズ3: テンプレート更新の下流追従）

## Context

#153 のプラットフォームエンジニアリング成熟度評価で、本リポジトリは golden path テンプレートとして
9.0/10 だが IDP（組織展開）としては 3.5/10 という結果が出た。ギャップの本質は「良いものが1リポジトリ
に閉じており、第2の消費者が自走できない」こと。現状、この golden path（FastAPI + Vue3 + Terraform +
3層品質ゲート + SDD 運用）を新プロジェクトで使うには「リポジトリを fork/コピーして手で改名・調整する」
しかない。

Epic #300 はこれを3フェーズで解消する計画で、フェーズ1「配布可能にする」の柱の一つが本 ADR の対象で
ある #294（スキャフォールドCLI導入）。フェーズ3の #298「テンプレート更新の下流追従（drift 対策）」
が既にロードマップに組み込まれていることから、単発生成だけでなく生成後の追従（テンプレート側の更新を
既存の生成済みプロジェクトへ再適用できること）を要件に含める必要がある。

検討した3案（#294 本文で指定された比較対象）:

1. **cookiecutter** — Jinja2 ベースのテンプレートエンジン。デファクトスタンダードで既存テンプレート
   資産・ドキュメントが豊富。ただし本体には生成後の更新追従機能がなく、`cruft`（サードパーティ）を
   併用して初めて `cruft update` による追従ができる。追従はテンプレート側との diff 適用で、機構が
   2ツールにまたがる。
2. **copier** — 同じく Jinja2 ベースだが、`copier update` による生成後プロジェクトへの追従が本体の
   一級機能。生成時に `.copier-answers.yml`（テンプレートバージョン・回答した変数値）を生成先に
   記録し、テンプレート更新時にそれを基準にした差分適用（3-way merge 相当）を行う。migrations
   フック（バージョン間の移行スクリプト）も持つ。cookiecutter と比べるとテンプレート資産の絶対数は
   少ないが、活発にメンテナンスされている。
3. **GitHub template repo（Use this template）+ 独自 bootstrap workflow** — 追加ツール依存なしで
   始められる。ただし「Use this template」自体はプレースホルダ置換を行わないため、生成後にリポジトリ
   名・変数を置換する独自スクリプト（sed 等）を書く必要がある。生成後の追従機構は存在せず、テンプレート
   側の更新を消費側へ反映するには、diff の手動確認・cherry-pick 相当の作業を毎回作り込む必要がある。

## Decision

**copier を採用する。**

- 決め手は #298 で計画されている「テンプレート更新の下流追従」が cookiecutter・GitHub template repo
  案では標準機能でカバーされず、copier のみが `copier update` として一級機能で持っている点。
  cookiecutter + cruft の組み合わせでも同等のことは可能だが、ツールを2つ運用するより copier 単体の
  方が学習コスト・保守コストが低いと判断した。
- GitHub template repo 案は初期コストの低さでは優れるが、追従機構を自前で作り込むことになり、それは
  実質的に copier が既に解決している問題を再実装することになるため却下する。
- テンプレート化の対象範囲（変数設計・置き場所の判断・生成物検証CI・README整備）は本 ADR のスコープ
  外とし、#294 の後続タスク・focused PR で個別に扱う。

## Consequences

- **良い面:**
  - `copier update` によって、フェーズ3の #298（drift 対策）・#299（JAWS-UG 磐田ハンズオンでの実証）
    をツール追加なしでそのまま実行できる。
  - Jinja2 ベースのテンプレート構文はエコシステムで広く知られており、学習コストが低い。
  - migrations フックにより、将来テンプレートの構造変更（ディレクトリリネーム等）を生成済みプロジェクト
    に安全に反映する手段を持てる。
- **トレードオフ・新たに生じる負担:**
  - `copier` は Python 製の追加 CLI 依存になる。devcontainer には未導入（`uv tool install copier` 等
    での導入が必要）。生成作業を行う第三者にも copier のインストールを前提としたドキュメントが要る。
  - cookiecutter と比べて公開テンプレート資産・Stack Overflow 等の情報量が少なく、複雑なテンプレート
    ロジックを書く際に参照できる実例が限られる。
  - `.copier-answers.yml` が生成先リポジトリに常駐するファイルとして増える。消費者がこのファイルの
    意味（テンプレートバージョン管理用）を理解している必要があり、README での説明が要る。
  - `copier update` の3-way merge は、消費側でテンプレート由来ファイルを大きく改変した場合にコンフリクト
    を起こし得る。コンフリクト解消の運用ルール（README または docs/ への追記）を #294 の後続タスクで
    整備する必要がある。
- **再検討トリガー:** #298（drift 対策の実運用）または #299（実証ハンズオン）で `copier update` の
  3-way merge が実用に耐えないレベルでコンフリクトを頻発させると判明した場合、cookiecutter + cruft、
  または生成後追従を諦めて GitHub template repo 案へ切り替えることを再検討する。
