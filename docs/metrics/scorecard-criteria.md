# プラットフォーム成熟度スコアカード 評価基準

**Status:** Accepted（2026-07-15）
**Related:** [#297](https://github.com/iwata-jawsug-jp/devcon/issues/297)、
`docs/proposal/platform-maturity-scorecard-criteria-proposal.md`（検討経緯）

このリポジトリのプラットフォームエンジニアリング成熟度を評価するための、10 軸の採点基準と
総合点の算出方法を定義する。第三者がこの基準を使って他リポジトリを評価できることを目標とする。

## 背景

[Qiita 記事](https://qiita.com/itouhi/items/039e1010dd25c70af5d9)（出典は
`.github/ISSUE_TEMPLATE/verification.md` の「関連するゴールデンパス軸」節）で、「Golden Path
テンプレートとしての完成度」「IDP（組織展開）としての完成度」をそれぞれ 10 点満点で自己評価し、
10 軸（各 5 点満点）で内訳を示す形式が使われてきたが、各軸の採点基準・総合点への集約方法は
文書化されていなかった。本書はそれを新規に定義したもの（検討経緯は `docs/proposal/` 参照）。

## 10 軸の 1〜5 点レベル定義

### 1. 開発環境の標準化

| 点  | 基準                                                                                                   |
| --- | ------------------------------------------------------------------------------------------------------ |
| 1   | 環境構築が手順書のみ（手動セットアップ）                                                               |
| 2   | Dev Container 等はあるが主要ツールのバージョンが未固定                                                 |
| 3   | Dev Container 化されているが、一部ツールのバージョンがローカルと CI でずれている                       |
| 4   | 主要言語・IaC ツールがバージョン固定され、`make dev` 一発で DB + backend + frontend が起動する         |
| 5   | 4 に加え、Dev Container のツールバージョン固定と CI 側のバージョン固定が一致することを機械検証している |

### 2. CI/CD

| 点  | 基準                                                                                                       |
| --- | ---------------------------------------------------------------------------------------------------------- |
| 1   | CI が存在しない、または手動実行のみ                                                                        |
| 2   | CI はあるが lint/test の一部のみ。main への直接 push でデプロイできる                                      |
| 3   | PR 時に lint/test/security が走るが、ローカル実行コマンドと処理内容が乖離している                          |
| 4   | 「ローカル green = CI green」が同一コマンドで保証され、main へのデプロイが OIDC + ロールで自動化されている |
| 5   | 4 に加え、reusable workflow 化・組織標準ガードレールとして他リポジトリから再利用可能な形で配布されている   |

### 3. 品質ゲート

| 点  | 基準                                                                                              |
| --- | ------------------------------------------------------------------------------------------------- |
| 1   | テスト・lint が存在しない、または恣意的に無効化できる                                             |
| 2   | テストはあるがカバレッジ閾値がなく、CI 上で skip 可能                                             |
| 3   | lint/test/型チェックの多層ゲートはあるが、対象範囲やバージョンが CI とローカルで食い違う          |
| 4   | lint/test/security の多層ゲートが CI で blocking、カバレッジ閾値がある                            |
| 5   | 4 に加え、実機 E2E スモーク等「デプロイが実際に動く」ことまで検証するゲートが blocking で存在する |

### 4. ドキュメント/ADR

| 点  | 基準                                                                                                            |
| --- | --------------------------------------------------------------------------------------------------------------- |
| 1   | README のみで、設計判断の記録がない                                                                             |
| 2   | 断片的な docs はあるが更新されておらず実装とズレている                                                          |
| 3   | docs はあるが「正」が一本化されておらず、複数箇所に重複記述がある                                               |
| 4   | ADR で意思決定が継続的に記録され、docs が実装規約の「正」として一本化されている                                 |
| 5   | 4 に加え、AI 向けファイル（`CLAUDE.md` 等）が docs からの薄い抽出として同期される仕組みとドリフト検出手順がある |

### 5. セキュリティガードレール

| 点  | 基準                                                                                                       |
| --- | ---------------------------------------------------------------------------------------------------------- |
| 1   | 長期的なクラウド認証情報がリポジトリまたは CI 変数に保存されている                                         |
| 2   | 一部 OIDC 化されているが、IAM 権限が広い（admin 相当）                                                     |
| 3   | OIDC ベースで最小権限を意識しているが、PR 用ロールが本番リソースを変更できる等の実質的な穴がある           |
| 4   | PR（plan）用ロールと deploy 用ロールが分離され、破壊的操作ができない最小権限になっている                   |
| 5   | 4 に加え、IAM Access Analyzer 等の静的ポリシー検証や secrets scanning が CI で blocking に組み込まれている |

### 6. 観測性/メトリクス

| 点  | 基準                                                                                              |
| --- | ------------------------------------------------------------------------------------------------- |
| 1   | ログ・メトリクスの収集がない                                                                      |
| 2   | アプリログはあるが構造化されておらず追跡できない                                                  |
| 3   | ログ収集はあるが、分散トレーシングや開発プロセスメトリクスがない                                  |
| 4   | 分散トレーシング（OTel 等）と DORA 基本指標（デプロイ頻度・変更リードタイム）が自動収集されている |
| 5   | 4 に加え、CFR/MTTR を含む DORA 4 指標が揃い、継続的にスコアカード等で推移が可視化されている       |

### 7. セルフサービス性

| 点  | 基準                                                                                               |
| --- | -------------------------------------------------------------------------------------------------- |
| 1   | 新規参加者が環境構築のたびに個別サポートを必要とする                                               |
| 2   | 手順書はあるが、複数ステップを手動で正確になぞる必要がある                                         |
| 3   | `make dev` 等でローカル起動はできるが、新規プロジェクトの立ち上げには手動コピー・改名が必要        |
| 4   | スキャフォールド CLI 等でコマンド一発で命名済み・CI 有効な新プロジェクトを生成できる               |
| 5   | 4 に加え、第三者が実際に生成 〜 初コミット 〜 実機検証まで自走できたことが TTFC 等で実測されている |

### 8. API 契約管理

| 点  | 基準                                                                          |
| --- | ----------------------------------------------------------------------------- |
| 1   | フロントエンド/バックエンド間の型が手書きで、頻繁にズレる                     |
| 2   | OpenAPI 等のスキーマはあるが型生成が手動・不定期                              |
| 3   | 型生成コマンドはあるが、CI で生成物と実装の一致を検証していない               |
| 4   | OpenAPI スキーマから型を自動生成し、CI で生成物と実装の一致をチェックしている |
| 5   | 4 に加え、契約の破壊的変更が CI で検出・ブロックされる仕組みがある            |

### 9. インフラ堅牢化

| 点  | 基準                                                                                                    |
| --- | ------------------------------------------------------------------------------------------------------- |
| 1   | state 管理がない、または誰でも本番リソースを直接変更できる                                              |
| 2   | IaC 化されているが、plan/apply 権限の分離がない                                                         |
| 3   | plan/apply 分離はあるが、WAF・KMS 暗号化・シークレットローテーション等の防御層が不足                    |
| 4   | state 保護（バージョニング + 暗号化 + 削除防止）・シークレット管理・plan/apply 最小権限分離ができている |
| 5   | 4 に加え、Policy as Code（conftest/OPA 等）で Terraform plan が機械的にポリシー検証されている           |

### 10. 組織スケーラビリティ

| 点  | 基準                                                                                                    |
| --- | ------------------------------------------------------------------------------------------------------- |
| 1   | 他チームがこの仕組みを使うには、リポジトリ全体をコピーして手動調整するしかない                          |
| 2   | 一部モジュール（Terraform 等）は切り出されているが、バージョンタグ付き配布がない                        |
| 3   | スキャフォールド CLI やモジュール配布はあるが、テンプレート更新を下流に追従させる仕組みがない           |
| 4   | タグ付き配布された共有モジュール・スキャフォールド CLI があり、第三者が自力で新プロジェクトを生成できる |
| 5   | 4 に加え、テンプレート更新の下流追従（drift 対策）の仕組みがあり、複数リポジトリのフリート運用ができる  |

## 10 軸 → 2 総合点への集約方法

10 軸は「単一リポジトリとしての作り込みの質」を測るものと「他チームが自力で展開・運用できるか」を
測るものに分かれる。[Epic #300](https://github.com/iwata-jawsug-jp/devcon/issues/300) が「IDP スコアの
支配的ボトルネックはセルフサービス性・組織スケーラビリティ」と明言していることを根拠に、次のとおり
割り当てる。

- **Golden Path テンプレート完成度**（8 軸）: 開発環境の標準化・CI/CD・品質ゲート・ドキュメント/ADR・
  セキュリティガードレール・観測性/メトリクス・API 契約管理・インフラ堅牢化
- **IDP（組織展開）完成度**（2 軸）: セルフサービス性・組織スケーラビリティ

各総合点 = 「割り当てられた軸の 5 点満点の平均 × 2」（= 10 点満点）。

> **v1 の割り切り**: CI/CD・ドキュメント/ADR・セキュリティガードレールは実際には IDP 側にも間接的に
> 効く（例: #295 の reusable workflow は他リポジトリからの再利用が主目的）。過去の定性評価の実績値を
> 再現するための重み付けチューニングは行わず、まず「主要因のみの単純平均」で運用を始め、乖離が大きい
> 場合に按分重み（0.5 倍等）を導入する改訂を検討する。

## 初期ベースライン（2026-07-15 時点）

| 軸                       | 点  | 根拠                                                                                                                                                                     |
| ------------------------ | --- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 開発環境の標準化         | 4   | Dev Container + `make dev` あり。CI とのバージョン一致検証は未整備                                                                                                       |
| CI/CD                    | 5   | #295 で reusable workflow 化・組織標準ガードレール配布済み                                                                                                               |
| 品質ゲート               | 5   | 3 層ゲート + ADR-0008 live-smoke E2E が blocking                                                                                                                         |
| ドキュメント/ADR         | 4   | ADR 運用・`docs/` 一本化あり。CLAUDE.md 同期のドリフト検出は手順化のみで自動化未実装                                                                                     |
| セキュリティガードレール | 4   | OIDC・plan/apply 分離済み。ADR-0009 で IAM Access Analyzer 導入済み                                                                                                      |
| 観測性/メトリクス        | 4   | ADR-0007 OTel・#237 で DORA 基本指標収集済み。CFR/MTTR は #42 完了後                                                                                                     |
| セルフサービス性         | 4   | #294 スキャフォールド CLI あり。TTFC 実測は #440（第2消費者実証）で確認済み                                                                                              |
| API 契約管理             | 3   | `make gen-types` はあるが、CI で生成物と実装の一致を検証するステップが現状ない（#297 項目1〜4のスコアカード自動化スクリプト作成時に判明、当初4と見積もっていたのを修正） |
| インフラ堅牢化           | 3   | state 保護・plan/apply 分離はあるが、Policy as Code（#296）は未着手                                                                                                      |
| 組織スケーラビリティ     | 3   | スキャフォールド CLI（#294）はあるが、共有モジュール配布（#239）・drift 追従（#298）は未完了                                                                             |

- **Golden Path テンプレート完成度**: (4+5+5+4+4+4+3+3) / 8 × 2 = **8.0 / 10**
- **IDP（組織展開）完成度**: (4+3) / 2 × 2 = **7.0 / 10**

このベースラインは `docs/metrics/scorecard/catalog.json` に人が設定する宣言スコアの初期値として
反映済み。自動生成スコアカード（#297 項目1〜4、[ADR-0014](../adr/0014-platform-maturity-scorecard-automation.md)）が
`docs/metrics/scorecard/` に履歴として記録する。
